Overview

As the book says, "black box" testing is when we devise tests based on the expected behavior of our code (we don't have any information about how the code is implemented—we don't know what's in the box). "White box" (or "glass box") testing is only possible when we do have access to the implementation: we can use that code to structure our tests, with the goal of making sure that all the code is "exercised" by the test.

Testing: FizzBuzz

There's a famous problem in the programming world called FizzBuzz. It's a simple problem: print out the numbers from 1 to 100, but everytime you hit a number divisible by 3, print "Fizz" instead. Every time you hit a number divisible by 5, print "Buzz" instead. If you hit a number divisible by 3 AND 5, print "FizzBuzz." Imagine your colleague has written a method public String fizzBuzzer(int x), which returns "-" if x is less than 1, and the string that FizzBuzz should print for x. Write a sequence of tests for the fizzBuzzer method.

Testing: Supermarket

Supermarket pricing can be tricky: some items have fixed prices, some have special prices if you buy multiples. For example, if a market is selling A, B, C, and D products, they might have pricing rules like this: 

  Item   Unit      Special
         Price     Price
  --------------------------
    A     50       3 for 130
    B     30       2 for 45
    C     20
    D     15

Our checkout system can accept items in any order, so that if we scan a B, an A, and another B, we’ll recognize the two B’s and price them at 45 (for a total price so far of 95).

Suppose we have a scan method that takes in a String representing a sequence of products (like "BAB" and returns an int representing the total price of those products). Write a sequence of tests for this method.
